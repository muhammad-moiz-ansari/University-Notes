Interactive guide to the four types of data dependence: true, anti, output, and input

What is it?

A true (or flow) dependence exists when one statement writes a value and a later statement reads it. The data literally flows from producer to consumer — this is a real, meaningful constraint.

Write → Read  ·  S1 δ S2
Example
S1:  a = 5        ! writes a
S2:  b = a + 1    ! reads a  ← depends on S1
S2 must execute after S1. Moving S2 before S1 would read an uninitialized or stale value.

In a loop context:

DO I = 1, N
  S1:  A(I+1) = B(I) + 1   ! writes A(I+1)
  S2:  C(I)  = A(I)         ! reads A(I)
ENDDO
! S1 on iteration i writes A(i+1)
! S2 on iteration i+1 reads A(i+1)  → loop-carried true dep
Can it be removed?
Cannot be eliminated

True dependences represent actual data flow — the ordering is real and necessary. You cannot remove it without changing the program's meaning. You can only work around it (e.g. by restructuring the algorithm).

What is it?

An anti-dependence is the reverse — one statement reads a variable and a later statement writes to it. The second statement must not write too early, or the first would read the wrong (new) value.

Read → Write  ·  S1 δ⁻¹ S2
Example
S1:  b = a + 1    ! reads a
S2:  a = 5        ! writes a  ← anti-dep on S1
If S2 ran first, S1 would read the new value of a (5) instead of the original. Order must be preserved.

In a loop context:

DO I = 1, N
  S1:  A(I)   = ... ! reads  (right-hand side)
  S2:  A(I+1) = ... ! writes A(I+1)
ENDDO
! S1 on iteration i+1 reads A(i+1)
! S2 on iteration i  writes A(i+1)  → anti-dependence
How to remove it
Can be eliminated via renaming

Introduce a new variable (rename) so the two statements no longer touch the same location:

Before
DO I = 1, N
  A(I) = A(I+1)
ENDDO
After (renaming)
DO I = 1, N
  A2(I) = A(I)
ENDDO
DO I = 1, N
  A(I) = A2(I+1)
ENDDO

Now the read and write target different arrays — no conflict, loop can be parallelized.

What is it?

An output dependence occurs when two statements both write to the same location. The last write must win — so their order matters.

Write → Write  ·  S1 δ⁰ S2
Example
S1:  a = 3    ! writes a
S2:  a = 7    ! writes a again ← output dep on S1
S3:  b = a    ! reads a — must get 7, not 3
If S2 ran before S1, then after both execute, a = 3 instead of 7. Any later read gets the wrong value.

In a loop context:

DO I = 1, N
  S1:  A(1) = B(I)   ! always writes A(1) — same location every iteration
ENDDO
! Every iteration writes A(1) → output dependence across all iterations
How to remove it
Can be eliminated via renaming

Give each write its own unique destination, then merge at the end if needed:

Before
S1:  a = 3
S2:  a = 7
S3:  b = a
After (renaming)
S1:  a1 = 3
S2:  a2 = 7
S3:  b  = a2  ! use last write

S1 and S2 now write to different variables — they can execute in any order or in parallel.

What is it?

An input dependence occurs when two statements both read from the same location. Since neither writes, there is no conflict — neither can corrupt the other's value.

Read → Read  ·  (not a true dependence)
Example
S1:  b = a + 1    ! reads a
S2:  c = a * 2    ! reads a
Both read the same value. No matter what order they run in, both get the correct value of a.

In a loop context:

DO I = 1, N
  S1:  b = a + I
  S2:  c = a * I    ! both just read a — no conflict
ENDDO
Does it need to be removed?
No action needed — already safe

Input dependences place no ordering constraint on execution. Both statements can run simultaneously without any risk. The only concern is cache performance — two reads to the same location is actually good for locality. No removal needed.
